# D-ACW-047 — v0.9.7: CLAUDE.md becomes thin pointer; auto-load moves to SessionStart hook

## Context

The `tools/scaffold-instance.py` scaffolder hardcoded nine `@`-imports into the generated `CLAUDE.md` (research/01-problem-framing.md, decisions/decision-log.md, rules/instance-hard-rules.md, tasks-status.md, glossary.md, research/evolution.md, research/sources.md, research/research-state.yaml, incidents.jsonl). The instance's `acw-state.yaml::auto_load_at_session_start` only earned four of those nine per `rules/auto-load-discipline.md`. Every freshly-scaffolded instance therefore began structurally non-compliant: CLAUDE.md auto-loading five files that hadn't earned the slot.

Drift surfaced on `Projects/cs-ops-spec/` (2026-05-13) immediately after scaffold. Operator caught it during the first session.

Beyond the immediate drift bug, the deeper problem: CLAUDE.md was carrying two concerns at once. It was simultaneously (a) the Claude Code-specific entry point pointing at the vendor-neutral AGENTS.md, and (b) the place where Claude Code's `@`-import mechanism listed the auto-load files. The two concerns are coupled only because Claude Code happens to read `@`-imports from CLAUDE.md. The coupling produces drift surface and violates the vendor-neutral spirit of AGENTS.md.

## Decision

CLAUDE.md becomes a one-line pointer (`See AGENTS.md.`). Auto-loading moves to a Claude Code SessionStart hook.

### Shape

1. `CLAUDE.md` (generated by scaffolder) — exactly `See AGENTS.md.\n`. No `@`-imports.
2. `.claude/settings.json` (generated by scaffolder) — registers SessionStart hook for matchers `startup|resume|clear`.
3. `.claude/hooks/load-context.py` (generated by scaffolder) — stdlib-only Python script. Reads `acw-state.yaml::auto_load_at_session_start`, concatenates the named files, emits `{"hookSpecificOutput": {"hookEventName": "SessionStart", "additionalContext": "<contents>"}}` to stdout. Claude Code injects `additionalContext` into context as a user message.
4. `AGENTS.md` directive 7 — rewritten to point at the SessionStart hook as the Claude Code implementation. Adds **Auto-load (Resource / When / Why)** table and **What NOT to Load** section, both adopted from JEVanClief's Interpreted Context Methodology pattern.
5. `rules/auto-load-discipline.md` — adds "Implementation: SessionStart hook" subsection naming the hook path, runtime contract, and why-this-shape rationale.

### Why SessionStart hook over alternatives

Researched 2026-05-13 (parent session). Five mechanisms surveyed: SessionStart hook (selected), path-scoped rules in `.claude/rules/`, AGENTS.md @-import or symlink from CLAUDE.md, auto-memory at `~/.claude/projects/.../memory/`, shared rules via `~/.claude/rules/` symlinks. SessionStart hook is the only mechanism that (a) is stable + documented, (b) lets the manifest stay the single source of truth (hook reads `acw-state.yaml` at runtime), (c) keeps CLAUDE.md trivially one-line, (d) is portable to other vendor hosts that support equivalent session-start hooks.

### What this does NOT change

- Per-prompt context cost. Both `@`-imports and SessionStart hooks load file content once at session start; the content rides along in every subsequent prompt's context window either way. The hook does not reduce per-prompt token cost. What reduces per-prompt cost is a smaller auto-load list (governed by `rules/auto-load-discipline.md`) and Anthropic prompt caching — neither is affected by this decision.
- The auto-load discipline gate. Every entry in `acw-state.yaml::auto_load_at_session_start` still requires `path` + `claim` + `earned_by`. The discipline is enforced at audit time, not at load-mechanism time. Bare-path legacy entries remain valid and surface for review at audit.
- AGENTS.md directives 1-6 and 8. Unchanged.

## Consequences

- Scaffold tool (`tools/scaffold-instance.py`) modified to emit the new shape. Pre-v0.9.7 instances scaffolded with the old shape continue to function — `@`-imports in their `CLAUDE.md` still load — but `/acw-instance upgrade` will propose migration to the new shape (CLAUDE.md trim + .claude/hooks/load-context.py + .claude/settings.json).
- `cs-ops-spec` instance backfilled in the same execution that ships this decision. CLAUDE.md trimmed to one line; `.claude/hooks/load-context.py` + `.claude/settings.json` written; AGENTS.md regenerated to match canonical.
- Vendor-neutrality of the ACW workspace shape is now structurally honored. Adding a Codex or Gemini host adapter is a matter of writing an equivalent SessionStart hook for that host that points at the same `acw-state.yaml` — no AGENTS.md changes needed.
- The Resource/When/Why table and What NOT to Load section adopt patterns from JEVanClief's Interpreted Context Methodology (research conducted 2026-05-13). Not borrowed: the L0/L1/L2 routing hierarchy, stage contracts, 4-room cap — those are orthogonal to load discipline and not earned by current ACW evidence.

## Migration

Pre-v0.9.7 instances:

1. Replace `CLAUDE.md` body with literal `See AGENTS.md.\n`.
2. Write `.claude/settings.json` with SessionStart hook config (see scaffold-tool template).
3. Write `.claude/hooks/load-context.py` (copy from ACW canonical at `tools/templates/load-context.py.tmpl`).
4. Update AGENTS.md directive 7 + add Resource/When/Why and What NOT to Load sections (copy from ACW canonical).
5. Smoke test: open Claude Code in the instance, verify additionalContext loads at session start.

`/acw-instance upgrade` automates steps 1-4 when run against a pre-v0.9.7 instance. The audit verb's "Host entry file shape (Claude Code) — v0.9.7 audit" pass (in `skills/acw-instance/references/audit.md`) detects pre-v0.9.7 shape and emits plan rows; the upgrade verb's "Host entry file migration (Claude Code) — v0.9.7" pass (in `skills/acw-instance/references/upgrade.md`) executes them under the standard plan-approval gate, including a post-migration hook smoke test. Trigger is filesystem-based: `CLAUDE.md` presence at workspace root signals Claude Code host.
