# Runbook — Migrating a downstream instance to v0.9.0 auto-load discipline

Operator how-to for applying ACW v0.9.0's auto-load discipline to an existing downstream instance (`cs-copilot`, `gsg-copilot`, `_Command`, or any other instance scaffolded before v0.9.0).

**What this runbook does:** in a single audit-then-upgrade pass, lands `rules/auto-load-discipline.md` AND demotes the instance's bloated auto-load list.

**What this runbook is not:** an automated migration. ACW's discipline says "list curation stays operator-driven per workspace." This is the walkthrough for that human-in-the-loop pass.

**Estimated time:** 5–10 minutes per instance.

---

## Pre-flight

Before running this runbook on any instance, confirm:

- ACW canonical is at `0.9.0` or later (`grep version: ~/projects/acw/acw-state.yaml`).
- The instance has a clean working tree (`git status` shows nothing uncommitted) — so the migration can be reverted as a single commit if needed.
- `gh` CLI is authenticated, OR `GITHUB_TOKEN` env var is set. The audit verb fetches canonical from GitHub.
- The operator has 5–10 minutes uninterrupted. Plan-approval gates are operator-blocking.

---

## Step 1 — Run `/acw-session start` in the instance

```
cd <instance path>
# Open a Claude Code session in this workspace
# Run: /acw-session start
```

Expected output: drift alert mentioning `0.9.0` recommendations not yet reconciled, including `rules/auto-load-discipline.md` (earned in 0.9.0). The instance's `last_reconciled_version` will be a value < `0.9.0`.

If no drift alert fires for v0.9.0 entries, the instance is already current — skip to Step 4.

---

## Step 2 — Run `/acw-instance audit`

```
/acw-instance audit
```

Audit fetches canonical from GitHub (including `rules/auto-load-discipline.md`) and walks the instance against it. The output combines the standard registry-gap rows with the new auto-load discipline section in one plan.

Look for the "Auto-load discipline" section:

```
Auto-load discipline (vs rules/auto-load-discipline.md):
  Current entries: <N>   form: <all-structured | all-legacy | mixed>
  Verdicts:
    KEEP                           : <N>
    KEEP (migrate-to-structured)   : <N>
    KEEP (instance-specific)       : <N>
    DEMOTE                         : <N>
    REVIEW                         : <N>
  Per-entry detail:
    <path>   form: <...>   verdict: <...>   reason: <...>
```

**Likely findings on a downstream instance scaffolded before v0.9.0:**

- 4 entries verdict `KEEP (migrate-to-structured)` — the four canonical recommendations with bare-path form.
- 0–4 entries verdict `DEMOTE` — `rules/manifest-discipline.md`, `rules/instance-current-manifest.md`, `rules/multi-instance-topology.md`, `incidents.jsonl` if present.
- 0–N entries verdict `REVIEW` — instance-specific additions the audit can't classify automatically.

The plan also includes the registry-gap rows (write-canonical for the discipline rule, reshape for the instance-current-manifest cache, version bump).

---

## Step 3 — Run `/acw-instance upgrade`

```
/acw-instance upgrade
```

Executes the audit's plan in one approval gate. The verb will:

1. Run pre-flight safety checks (offer pre-migration commit if uncommitted changes).
2. Print the migration plan — review the auto-load section specifically.
3. Apply in bulk:
   - Write `rules/auto-load-discipline.md` to the instance.
   - Overwrite local `rules/instance-current-manifest.md` cache with fetched canonical.
   - Convert KEEP entries to structured form using canonical claims (no operator input for the four canonical recommendations).
   - Remove `DEMOTE` entries from `auto_load_at_session_start`. Files stay in the workspace; only the auto-load reference drops.
   - Prompt interactively for each `REVIEW` entry — operator chooses keep-with-claim, demote, or skip.
   - Update host entry files (e.g., `CLAUDE.md` `@`-imports) to mirror the new auto-load list.
4. Bump `last_reconciled_version` to `0.9.0`.
5. Log a decision-log entry recording the reconciliation.

**Before approving, confirm:**

- KEEP entries match what you want to stay auto-loaded.
- DEMOTE entries are paths whose consumer-skill loads them on demand (no agent will be missing them in chat).
- For REVIEW entries: pick `k` only if you have a specific claim ready; otherwise default to `d` (DEMOTE).

Approve `[y]`.

---

## Step 4 — Verify the lean state

After the upgrade lands:

```
# Quick check
python ~/projects/acw/tools/manifest.py load <instance>/acw-state.yaml auto_load_at_session_start
```

Expected output: a Python list of 4 paths (or whatever lean count you approved).

Open a fresh Claude Code chat in the workspace. The context-usage panel (`/context` if available, or the IDE's panel) should show `Memory files` dropping by ~25–30k from baseline.

If the new context-load number isn't lower:

- Check `CLAUDE.md` — `@`-imports mirror the new auto-load list?
- Check `acw-state.yaml::auto_load_at_session_start` — structured form, 4 entries?
- The Claude Code prompt cache may take 1–2 sessions to refresh; the second cold-start session shows the new baseline.

---

## Step 5 — Commit

```
cd <instance path>
git add -A
git commit -m "feat: migrate to ACW v0.9.0 — auto-load discipline applied"
```

The pre-migration safety commit (created by `/acw-instance upgrade` in Step 3 if you had uncommitted changes) plus this migration commit bracket the entire operation. `git revert <migration-commit>` rolls back if anything looks wrong.

---

## Per-instance notes

### `cs-copilot`

Substrate-shaped, was registered in v0.4.0+. Likely outcome: 4 demotions (the same 4 that ACW canonical demoted), no REVIEW entries. Quick pass.

### `gsg-copilot`

The original v0.2.0-rc1 source instance. Older registration; may have legacy bare-path everywhere. Audit may surface multiple `KEEP (migrate-to-structured)` rows.

### `_Command`

Migrated to ACW v0.7.0 in Session 12. Should be at `last_reconciled_version: "0.7.0"` or `"0.8.0"`. Audit will pick up the v0.8.0 + v0.9.0 deltas in one plan.

### Other workspaces

If a workspace has a custom auto-load list (not the canonical 4), the audit's REVIEW rows will surface them. The operator's job: declare a structured `claim` for each one that should stay, or accept the demote recommendation.

---

## When this runbook does NOT apply

- Workspace is not an ACW instance — `tools/scaffold-instance.py` first.
- Workspace is at `last_reconciled_version: "0.9.0"` or later — already migrated.
- Instance has `is_canonical_source: true` — that's ACW itself or a sibling canonical-publishing meta-instance; v0.9.0 was applied to ACW directly in the v0.9.0 ship session.

---

## What to do if something goes wrong

| Symptom | Diagnosis | Fix |
|---|---|---|
| `gh: command not found` | gh CLI missing | Install gh, authenticate, re-run |
| Audit reports zero auto-load rows | Block already structured-form lean | No action; instance was already current |
| `git revert` of migration commit | Operator unhappy with result | Run revert, investigate, file a decision-log entry, propose corrections to ACW canonical if the discipline itself is wrong |
| Drift alert persists after upgrade | `last_reconciled_version` didn't bump | Inspect `acw-state.yaml`; manually set if needed; investigate why the upgrade didn't bump it |
| Audit shows `REVIEW` for an entry the operator wants to keep | Need a structured claim | Add the entry as structured form in `acw-state.yaml::auto_load_at_session_start` with a one-line claim; re-audit; should now show `KEEP (instance-specific)` |
